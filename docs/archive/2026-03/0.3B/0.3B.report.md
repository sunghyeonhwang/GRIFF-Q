### 4.2 데이터베이스 설계

#### 테이블 1: `project_kickoffs`

```sql
CREATE TABLE project_kickoffs (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id uuid NOT NULL REFERENCES projects(id) ON DELETE CASCADE UNIQUE,
  objective text,
  scope text,
  constraints text,
  success_criteria text,
  kickoff_date date,
  meeting_notes text,
  meeting_attendees uuid[] DEFAULT '{}',
  meeting_decisions jsonb DEFAULT '[]',
  status text NOT NULL DEFAULT 'draft' CHECK (status IN ('draft', 'in_progress', 'completed')),
  completed_at timestamptz,
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now()
);
```

**인덱스**: idx_kickoffs_project (project_id)
**RLS**: SELECT (모든 인증 사용자) / INSERT (모든 인증 사용자) / UPDATE (모든 인증 사용자) / DELETE (super/boss/manager만)
**트리거**: update_updated_at (수정 시 updated_at 자동 갱신)

#### 테이블 2: `kickoff_checklist_items`

```sql
CREATE TABLE kickoff_checklist_items (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  kickoff_id uuid NOT NULL REFERENCES project_kickoffs(id) ON DELETE CASCADE,
  title text NOT NULL,
  description text,
  assignee_id uuid REFERENCES auth.users(id) ON DELETE SET NULL,
  is_completed boolean DEFAULT false,
  due_date date,
  sort_order integer DEFAULT 0,
  created_at timestamptz DEFAULT now()
);
```

**인덱스**: idx_checklist_kickoff (kickoff_id)
**RLS**: CRUD 모두 허용 (프로젝트 팀원 대상)

#### 테이블 3: `schedules`

```sql
CREATE TABLE schedules (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  title text NOT NULL,
  description text,
  category text NOT NULL DEFAULT 'other'
    CHECK (category IN ('vacation', 'salary_review', 'birthday', 'holiday', 'company', 'meeting', 'other')),
  start_date date NOT NULL,
  end_date date,
  is_all_day boolean DEFAULT true,
  start_time time,
  end_time time,
  color text,
  created_by uuid NOT NULL REFERENCES auth.users(id),
  is_public boolean DEFAULT true,
  target_user_id uuid REFERENCES auth.users(id) ON DELETE SET NULL,
  created_at timestamptz DEFAULT now()
);
```

**인덱스**: idx_schedules_dates, idx_schedules_created_by, idx_schedules_category
**RLS**:
- SELECT: is_public=true OR created_by=me OR target_user_id=me
- UPDATE: created_by=me OR (super/boss/manager)
- DELETE: 동일

## Lessons Learned

### 7.1 What Went Well

1. **설계와 구현의 높은 일치**
   - Design 문서가 매우 상세하여 구현 방향 명확
   - TypeScript 타입 정의가 선행되어 런타임 에러 최소화
   - 96% → 99%+로 개선 용이

2. **점진적 Gap 수정 프로세스**
   - Gap Analysis 문서로 명확한 이슈 도출
   - Major/Minor 분류로 우선순위 설정
   - 2회 반복(Iteration)으로 100% 완성

3. **기존 컴포넌트 재사용**
   - @dnd-kit (기존 Kanban에서 사용) → 킥오프 체크리스트에 재사용
   - ProjectMilestoneTimeline 컴포넌트 → 킥오프-마일스톤에 재사용
   - 코드 중복 최소화, 일관성 유지

4. **RLS 정책의 명확한 설계**
   - project_kickoffs: 팀원 전체 읽기, super/boss/manager만 삭제
   - schedules: is_public으로 공개/비공개 구분, 작성자/대상자만 접근
   - 권한 체계 구축 용이

### 7.2 Areas for Improvement

1. **모바일 Sheet UI의 초기 누락**
   - Design에 "모바일: Sheet로 표시"라고 명시했으나 구현 놓침
   - 원인: Desktop-first 개발 후 모바일 검증 순서로 진행
   - 개선: 향후 Design 문서의 "모바일 특수 사항" 섹션 별도 강조

2. **DatePicker 입력 필드의 선택적 구현**
   - due_date는 Server Action에서 지원하나 UI 폼에 누락
   - 원인: 기능 분석 시 "optional" 필드로 판단
   - 개선: 설계 단계에서 "선택적이어도 UI 폼에는 포함" 명시

3. **CSS 컬러 시스템의 버전 관리**
   - Design: hsl() 변수 / Implementation: oklch() (Tailwind v4)
   - 원인: Tailwind v4 마이그레이션 중 진행된 구현
   - 개선: Design 문서의 "CSS 버전" 명시 (hsl v3 / oklch v4)

4. **FullCalendar 번들 최적화**
   - 초기 번들 크기 예상보다 큼 (~250KB)
   - 동적 import 및 tree-shaking 충분했나 검토 필요
   - 개선: v0.4에서 성능 프로파일링 수행

### 7.3 To Apply Next Time

1. **Design Phase에서 모바일 UI 상세 정의**
   - "데스크톱 사이드바 숨김 → 모바일 Sheet 표시" 같은 반응형 기술 명시
   - Design 템플릿에 "Mobile Breakpoint (lg/md/sm)" 섹션 추가

2. **선택적 필드의 UI 포함 여부 명확화**
   - Server Action에서 지원하는 필드 = 반드시 UI 폼에 포함
   - "Optional이어도 입력 필드는 표시, 사용자에게 선택권 제공" 원칙

3. **초기 설계 단계의 버전 호환성 확인**
   - 신규 기능 설계 시 프로젝트의 현재 "기술 스택 버전" 확인
   - CSS/라이브러리 버전 명시 (Tailwind v3 vs v4 등)

4. **QA 단계에서 4가지 조합 필수화**
   - Light/Dark × Desktop/Mobile 4가지 조합을 CI/CD에 자동화
   - 모바일 우선 테스트 (넓은 화면보다 제약 조건이 많음)

5. **Gap Analysis를 매 반복마다 즉시 재수행**

   - "이전 96% → 현재 99%"처럼 진행 상황을 시각화
